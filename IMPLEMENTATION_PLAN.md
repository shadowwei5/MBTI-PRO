> 版本: v1.0 | 日期: 2026-06-10 | 基于：项目计划书 + 2份调研报告 + 网络调研

## 执行摘要

本项目"MBTI-PRO"是一个创新型在线人格测评平台。核心创新在于将传统MBTI的四维**二分法（16型）升级为四维三级分类（81型）**，在每个维度引入中间值（A/B/C/D），解决传统二分法对"中间倾向"用户不友好的问题。

**核心差异化优势：**
- 81型 vs 传统16型 → 精细度提升5倍
- 80题李克特量表 + 20题客观推理题 → 混合测量降低主观偏差
- 免注册 + 匿名设计 → 零门槛+隐私保护
- 原生中文 + 本土化内容 → 更好的中国用户体验

**预计开发周期：10周 | 团队规模：4-6人 | 初始月运营成本：¥230-430**

## 一、项目背景与分析

### 1.1 市场机会

基于调研分析，当前MBTI测试市场存在以下痛点：

| 痛点 | 传统MBTI | 16personalities | 才储64型 | **MBTI-PRO方案** |
|------|---------|----------------|---------|-----------------|
| 二分法标签化 | 严重 | 部分解决(A/T) | 维度过多 | **四维三级，精校准** |
| 主观偏差 | 纯自评 | 纯自评 | 纯自评 | **客观题+主观题混合** |
| 中文适配 | 翻译腔 | 翻译腔 | 本土化好 | **原生中文深度适配** |
| 使用门槛 | 付费/注册 | 免费 | 部分免费 | **全功能免费+免注册** |
| 精细度 | 16型 | 32型 | 64型 | **81型** |

### 1.2 用户需求验证

- 2025年中国MBTI用户约5800万，学生群体占23%
- 63%的用户抱怨MBTI报告千篇一律
- 43%的用户在重测时获得不同结果（二分法导致）
- 约35%的用户在某个维度上得分接近零（中间倾向）

**MBTI-PRO的81型分类直接解决了中间倾向用户的需求，潜在覆盖约2000万"中间型"用户。**

## 三、技术实施方案

### 3.1 技术栈确定

| 层级 | 技术选型 | 版本 |
|------|---------|------|
| 前端框架 | Vue 3 + Composition API | 3.4+ |

| ORM | Prisma | 5.8 |
| 数据库 | MySQL | 8.0 |
| 缓存 | Redis | 7.x |
| 部署 | Docker + Nginx | - |

### 3.2 核心算法实施

**评分引擎实现要点：**
1. 输入验证：100个答案完整性检查
2. 按维度分组：将答案映射到4个维度
3. 计分方向处理：正向题/反向题自动识别
4. 客观题校验：与正确答案比对
5. 三级判定：按阈值分类
6. 置信度计算：距离阈值的远近
7. 结果缓存：相同答案组合缓存评分结果

**关键代码位置：**
- `server/src/services/scoringService.ts` - 后端评分引擎
- `src/utils/scoring.ts` - 前端离线评分(备选)

### 3.3 数据库实施

**核心表：**
1. `questions` - 题库表（100条记录）
2. `personality_types` - 81型人格表（81条记录）
3. `test_records` - 测试记录表（匿名，持续增长）
4. `career_matches` - 职业匹配表（可选）
5. `admin_users` - 管理员表（可选）

**初始化数据量估算：**
- 100道题目 × 约2KB/题 = 200KB
- 81型描述 × 约1.5KB/型 = 121KB
- 总计初始数据 < 1MB

### 3.4 部署方案

## 四、内容实施方案

### 4.1 题库开发计划

**出题原则：**
1. 每道题只测一个维度一个子维度
2. 语言通俗，使用中国日常生活场景
3. 避免引导性和社会赞许性
4. 正向题和反向题数量平衡（各约50%）
5. 客观题难度逐步提升

**出题流程：**
```
命题 → 初审(内容质量) → 试测(5人) → 修订 → 终审 → 入库
```

### 4.2 81型描述文案开发

**策略：模块化 + AI辅助**

采用"12个维度模块 + 81个组合描述"的模块化方法：

**第一步：编写12个维度模块（第5周）**
- 每个维度3个取值方向 × 4个维度 = 12个模块
- 每个模块100-150字
- 工作量：约2天

**第二步：编写81个组合描述（第5-6周）**
- 使用AI(ChatGPT/Claude)批量生成初稿
- 人工逐条审核修订
- 重点关注32个"含1个中间值"的类型（核心差异化内容）
- 工作量：约8天（AI辅助可大幅提速）

**第三步：职业匹配数据（第6周）**
- 基于传统16型职业数据库
- 对中间值类型进行插值调整
- 每型推荐5-8个岗位，分高度契合/中度适配两级
- 工作量：约2天

### 4.3 视觉素材开发

**传统16型头像：**
- 参考社区成熟角色形象（如16personalities的小人）
- 重新绘制或AI生成 + 人工后期
- 保持原色系，统一风格

**新增65型头像：**
- 使用Midjourney/DALL-E/Stable Diffusion批量生成
- 每个头像生成3-5个候选，人工筛选
- 统一风格：扁平插画风、低饱和色系、中性友好

**头像规格：**
- 格式：WebP（主）+ PNG（降级）
- 尺寸：400×400px（@2x）
- 背景：透明

## 五、质量保证方案

### 5.1 测试策略

| 测试类型 | 覆盖率目标 | 工具 | 关键测试点 |
|---------|-----------|------|-----------|
| 单元测试 | ≥ 90% (评分引擎) | Vitest | 评分算法、阈值判定、置信度 |
| 集成测试 | ≥ 70% (API) | Vitest + Supertest | API端点、数据验证 |
| E2E测试 | 核心流程全覆盖 | Playwright | 首页→答题→提交→结果→分享 |
| 性能测试 | Lighthouse ≥ 90 | Lighthouse CI | 首屏加载、交互响应 |
| 安全测试 | OWASP Top 10 | 人工+自动化 | XSS、CSRF、注入 |
| 可访问性测试 | WCAG AA | axe-core | 键盘、读屏器、色彩对比 |

### 5.2 评分准确性质控

1. **算法正确性验证**：手工计算 vs 程序计算，100个测试用例交叉验证
2. **阈值合理性验证**：内测数据分布分析，确保三级各占合理比例
3. **重测一致性验证**：20人间隔2周重测，计算重测一致性系数
4. **边界值测试**：每种阈值附近（如E_I得分=16, 17, -17, -16）的判定准确性

### 5.3 用户测试

| 阶段 | 人数 | 目标 | 关键指标 |
|------|------|------|---------|
| 内部试测 | 5人 | 题目可读性、难度评估 | 完成率、平均答题时间 |
| 内测Alpha | 20人 | 全流程验证、算法校准 | 完成率、满意度、类型分布 |

## 六、风险管控方案

| 风险 | 等级 | 缓解方案 | 应急方案 |
|------|------|---------|---------|
| 题库质量不达标 | 高 | 邀请心理学专业顾问审核 | 先用28题简化版上线，后续迭代 |
| 81型描述文案延期 | 中 | AI批量生成+人工审核提速 | V1.0仅上线16型详细描述，81型用模块化拼接 |
| 客观题难度失控 | 中 | 先出30题，试测筛选20题 | 降低客观题权重或题目数量 |
| T/F阈值偏差大 | 中 | 内测数据分析后校准 | 发布时标注"Beta"版本，收集数据后调整 |
| 服务器成本超预算 | 低 | 优先纯静态方案 | 临时切换至纯前端+静态部署 |
| 法律合规风险 | 低 | 上线前法律审查+免责声明 | 暂缓企业版功能，仅做个人版 |
